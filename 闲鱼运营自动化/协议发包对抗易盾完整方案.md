## 第一章 协议发包到底是什么

### 1.1 概念对比

```
[传统真机方案]
真机 App
  → 易盾 SDK 自动采集 80+ 项指纹
  → POST 给易盾服务器
  → 易盾返回 deviceToken
  → 业务请求带上 deviceToken
  → 业务后端校验通过
  → 完成业务

[协议发包方案]
你的 Python/Go 脚本
  → 直接构造业务接口请求
  → 把"必要的风控字段"自己拼进去
  → POST 给业务后端
  → 业务后端校验通过
  → 完成业务
```

协议发包的本质是 **「跳过端,直接说人话」**。但因为风控存在,你必须证明"我是真的从一台真手机里说的"。

### 1.2 协议发包的优势

| 维度 | 真机方案 | 协议方案 |
|---|---|---|
| 单账号成本 | ¥1500 设备 + ¥30/月运营商 | ¥0(只算云端 IP+服务器) |
| 并发能力 | 1 设备 1 账号 | 几乎无限(看 IP 池) |
| 部署速度 | 物理调试,慢 | 部署即用,快 |
| 跨地理位置 | 需要不同地区 SIM 卡 | 换代理就行 |
| 维护成本 | 设备故障 / 系统升级 | 软件维护为主 |

### 1.3 协议发包的命门

> 易盾最关心的不是"你这次请求长什么样",而是 **"这台'设备'此刻是不是真的在跑 App"**。

协议方案天然没有"端在跑"这件事,所以对抗的核心就是 **让易盾后端误以为有一台真机在跑**。这是后续所有方案的底层逻辑。

### 1.4 协议发包不能做什么

诚实地说,有三件事协议方案永远做不到:

1. **绕过硬件级可信认证(KeyMint / Widevine L1)** — 闲鱼/小红书目前还没强制要求,但抖音、银行、政务类 App 已经在用,协议方案在这类场景直接死
2. **应对端上完整性校验** — 比如 App 启动会扫自己 APK 的 SHA256 上报,协议方案没有"自己"
3. **应对设备-账号长期绑定的风控** — 后端关联 5 个月的"该设备登过哪些号",协议方案的"虚拟设备"没有这种历史

→ 这就是为什么后面强烈推荐 **混合架构(方案 D)** 的根本原因。

## 第二章 协议方案下,易盾如何识别你

回到风控判定的 8 大维度,在协议层会暴露成以下信号:

### 2.1 八大判定维度的协议层暴露点

| 维度 | 真机怎么过 | 协议层的命门 |
|---|---|---|
| ID 一致性 | SDK 自己算 deviceToken | **deviceToken 你拿不到**,业务请求里这字段一假就死 |
| 硬件一致性 | 真机本来就一致 | 你必须**预生成完整画像**,字段相互绑定 |
| 系统一致性 | 真机本来就一致 | 同上,而且要绑定 SDK 版本 |
| App 环境 | 包路径正常 | 协议层无法伪装,但请求里有"包签名校验"字段 |
| 网络一致性 | IP 来自真机 | **代理出口 IP 必须与声称的设备地理位置一致** |
| 行为时间 | 用户真在操作 | **多请求之间的时间间隔必须合理** |
| 账号-号码历史 | 后端关联 | 后端关联,**端上无法对抗** |
| 异常环境 | TrickyStore 等 | 协议层不存在这一项 |

### 2.2 真正的三大难点

总结起来,协议方案的核心难点就三个:

#### 难点 1:**deviceToken 怎么来**
易盾的 deviceToken 是它**服务器签发**的,不是端上算的。所以你不能凭空造,只能:
- 让一台真机帮你算(方案 A、D)
- 自己模拟易盾 SDK 的上报协议拿(方案 B)
- 让云手机里的真易盾 SDK 算(方案 C)

#### 难点 2:**字段一致性**
80+ 字段不能各自随机,必须按"机型"为锚点强绑定。这是协议方案最累的工程量。

#### 难点 3:**协议层的"心跳上报"伪造**
易盾 SDK 不只在请求时上报,还有 5-10 分钟一次的"心跳"。如果你只在业务请求时调一次签名,但从不发心跳,后端会判定"该设备 SDK 死了 → 异常"。

## 第四章 方案 A:Sekiro RPC 签名服务(工业级首选)

### 4.1 整体架构

```
┌─────────────────────────────────────────────────┐
│  云端 Python 协议爬虫(可水平扩展)              │
│  - 自己拼业务请求                                │
│  - 维护设备画像 JSON 池                          │
│  - 维护代理 IP 池                                │
│  - 维护账号池                                    │
│  - 需要 deviceToken / sign 时:                  │
│      → HTTP 调 sekiro server                    │
└────────────────────┬────────────────────────────┘
                     │ HTTP RPC
                     ▼
┌─────────────────────────────────────────────────┐
│  Sekiro Server(Java,可单机或集群)             │
│  - 路由请求到注册的真机                          │
│  - 负载均衡                                      │
│  - 心跳监控                                      │
└────────────────────┬────────────────────────────┘
                     │ 长连接
        ┌────────────┼────────────┐
        ▼            ▼            ▼
   [真机1]       [真机2]      [真机N]
   闲鱼App      闲鱼App      闲鱼App
   Frida已注入  Frida已注入  Frida已注入
   sekiro-client 暴露关键签名/采集函数
```

**核心脚本 `sekiro_yidun_xianyu.js`**:

```javascript
// 1. 加载 sekiro client
function require_sekiro() {
    Java.perform(function () {
        var client = new SekiroClient(
            "ws://your-server:5621/business/register?" +
            "group=yidun-pool&" +
            "clientId=phone-001"
        );

        // === 暴露 1: 取易盾 deviceToken ===
        client.registerAction("get_yidun_token", function (req, resp) {
            Java.perform(function () {
                try {
                    var SmAntiFraud = Java.use("com.netease.mobsec.SmAntiFraud");
                    var token = SmAntiFraud.getDeviceId();
                    resp.success({ deviceToken: token, phoneId: "phone-001" });
                } catch (e) {
                    resp.failed("getDeviceId error: " + e);
                }
            });
        });

        // === 暴露 2: 闲鱼 mtop sign ===
        client.registerAction("xianyu_mtop_sign", function (req, resp) {
            Java.perform(function () {
                try {
                    var data = req.data;       // 业务方传入待签数据
                    var apiName = req.apiName; // 接口名
                    
                    var SecGuardMgr = Java.use(
                        "com.taobao.wireless.security.adapter.JNICLibrary"
                    );
                    // hook 调用真签名函数
                    var sign = /* hook 出来的 sign */;
                    resp.success({ sign: sign });
                } catch (e) {
                    resp.failed("sign error: " + e);
                }
            });
        });

        // === 暴露 3: 触发一次完整易盾上报(伪心跳)===
        client.registerAction("yidun_heartbeat", function (req, resp) {
            Java.perform(function () {
                // 调用易盾 SDK 内部的上报方法
                var WatchMan = Java.use("com.netease.mobsec.rjsb.watchman.WatchMan");
                WatchMan.report.call();
                resp.success({ ok: true });
            });
        });
    });
}

setImmediate(require_sekiro);
```

### 4.4 Python 端调用

```python
import requests
import time
import random

SEKIRO_HOST = "http://your-server:5620"
GROUP = "yidun-pool"

def call_sekiro(action, **kwargs):
    """调用 sekiro 暴露的函数"""
    url = f"{SEKIRO_HOST}/business/invoke"
    params = {"group": GROUP, "action": action, **kwargs}
    r = requests.post(url, params=params, timeout=10)
    j = r.json()
    if j.get("status") != 0:
        raise RuntimeError(f"sekiro {action} failed: {j}")
    return j["data"]

def get_yidun_token():
    return call_sekiro("get_yidun_token")["deviceToken"]

def xianyu_sign(api_name, data):
    return call_sekiro("xianyu_mtop_sign", apiName=api_name, data=data)["sign"]

def heartbeat():
    """模拟 SDK 心跳,每 5-10 分钟调一次"""
    return call_sekiro("yidun_heartbeat")

# === 业务请求示例 ===
def publish_xianyu_item(profile, account, item_data):
    api = "mtop.idle.publish"
    sign = xianyu_sign(api, item_data)
    token = get_yidun_token()
    
    headers = {
        "x-mtop-sign": sign,
        "x-device-id": token,
        "x-uid": account.uid,
        "user-agent": profile.ua,
        "x-app-version": profile.app_version,
        # ... 其他字段
    }
    
    r = requests.post(
        "https://acs.m.taobao.com/gw/" + api,
        headers=headers,
        json=item_data,
        proxies={"https": account.proxy},
        timeout=15
    )
    return r.json()
```

### 4.5 调度策略

一台真机不能无限出签名,建议:

```python
# Sekiro server 自带负载均衡,但要给单机限速
class PhonePool:
    def __init__(self):
        self.phones = ["phone-001", "phone-002", ...]
        self.last_call = {}  # phone -> timestamp
    
    def get_phone(self):
        # 取最久没用的
        now = time.time()
        candidates = sorted(
            self.phones,
            key=lambda p: self.last_call.get(p, 0)
        )
        for p in candidates:
            if now - self.last_call.get(p, 0) > 1.5:  # 单机 1.5s 限速
                self.last_call[p] = now
                return p
        time.sleep(1)
        return self.get_phone()
```

实测一台真机稳定出签名:**1000~3000 次/小时**(看 App 复杂度)。10 台真机够支撑 100 个云端账号的全天活动。


## 第五章 方案 B:纯协议模拟易盾上报

### 5.1 整体思路

不要任何真机,**自己实现一遍易盾 SDK 的协议**:

```
1. 静态逆向易盾 SDK(libDeviceFingerprint.so 等)
2. 还原它的:
   - 数据采集协议(80+ 字段如何编码成 protobuf/JSON)
   - 加密算法(通常是 AES + RSA + 自研混淆)
   - 上报接口(POST 到 lookup.dun.163.com 等)
3. Python 端:
   - 维护设备画像 JSON
   - 按照还原的协议,把假数据加密后 POST 给易盾服务器
   - 拿回 deviceToken
   - 用 token 去发业务请求
```

### 5.2 主要工作量

| 任务 | 工作量 | 难度 |
|---|---|---|
| 易盾 SDK 静态逆向 | 1-3 周 | 极高(VMP 加固 + 反调试) |
| 还原 80+ 字段编码格式 | 1 周 | 中 |
| 还原加密算法(AES key 提取) | 1-2 周 | 高 |
| 还原 RSA 公钥与签名流程 | 3-5 天 | 中 |
| Python 重写实现 | 1 周 | 中 |
| 上线测试 + 修字段 | 持续 | 中 |
| **每月维护(算法变了重做)** | 1-2 周 | 极高 |

### 5.3 优劣分析

**优点**:
- 极致并发,纯云端
- 无真机成本
- 可以批量部署到任意服务器

**缺点**:
- 易盾算法**每月都在更新**,你今天逆向完,下个月失效
- 易盾有强反逆向(VMP 加固 + 代码混淆 + 反调试 + 自校验)
- 一旦你的"假上报"被识别(后端有行为分析),整个 IP 段都封
- 需要**专职逆向工程师**长期维护
- 端上签名校验也要破(更难)

### 5.4 何时值得做

- 账号矩阵 > 10 万
- 有 2 人以上的逆向团队
- 业务长期稳定(不会过几个月就转方向)
- 能承受"算法更新期"几天的全停业损失

> 💡 **行业现状**:GitHub 上**没有完整开源的易盾协议实现**,能找到的都是过期、零散的、demo 性质代码。市面上"易盾签名 API"按调用计费的就是走这条路的玩家在卖。
>
> 个人/小团队**不建议自己做**这个,ROI 极低。

### 5.5 思路参考(高阶玩家速看)

如果你坚持要走这条路,关键步骤如下:

```
1. 拿到目标 App 的最新版 APK
2. unzip + 找出易盾 .so 文件
   → 通常在 lib/arm64-v8a/libNetEaseMobsec.so 等
3. IDA Pro 或 Ghidra 打开,找以下关键函数:
   - Java_xxx_collect    (字段采集入口)
   - Java_xxx_encrypt    (加密入口)
   - Java_xxx_report     (上报入口)
4. Frida hook 这几个函数,dump 输入输出对
5. 收集足够样本后,反推算法
6. 关键: AES key 通常是动态生成的,要从 RSA 解密流程逆推
7. Python 重写,用 cryptography 库实现等价加密
8. 用真机抓包对比,直到 Python 出的请求和真机一模一样
```

## 第六章 方案 C:云手机 + ADB 协议

### 6.1 整体架构

```
[云手机集群: Redroid/华为云手机/腾讯云手机/阿里云手机]
   ↓ 每台云手机装着闲鱼 App + 改机模块
   ↓
你的脚本通过 ADB shell input + Accessibility Service 操控
   ↓
易盾 SDK 在云手机里跑,采集到的是改机后的指纹
```

### 6.2 云手机选型

| 厂商 | 价格 | 改机难度 | 风控通过率 |
|---|---|---|---|
| 华为云手机 | ¥40/月 | 极难(只读 ROM) | 低(大批量被风控标记) |
| 阿里云手机 | ¥30/月 | 难 | 低 |
| 腾讯云手机 | ¥35/月 | 难 | 低 |
| 红手指 | ¥20/月 | 中 | 中 |
| 雷电云 | ¥25/月 | 易 | 中 |
| **自建 Redroid** | ¥80/月(GPU 服务器) | 易(可改 AOSP) | 中-高 |

### 6.3 自建 Redroid 流程

```bash
# 1. 准备一台 GPU 服务器(NVIDIA T4/A10 起步)
# 2. 装 Docker + nvidia-container-toolkit
# 3. 拉 redroid 镜像
docker pull redroid/redroid:14.0.0_64only-latest

# 4. 启动一个实例(可起多个)
docker run -d --rm --privileged \
    --name redroid-001 \
    --network=host \
    -v /data/redroid-001:/data \
    redroid/redroid:14.0.0_64only-latest \
    androidboot.redroid_gpu_mode=guest

# 5. ADB 连接
adb connect localhost:5555
```

### 6.4 改 AOSP 源码做指纹随机化(高阶)

这是 Redroid 方案的真正杀手锏:

```
target: frameworks/base
1. android.os.Build 的 SystemProperties → 改成读 /data/local/profile.json
2. android.provider.Settings$Secure 的 ANDROID_ID → 同上
3. android.telephony.TelephonyManager → 同上  
4. WifiInfo 的 MAC/BSSID → 同上
5. 重新编译 AOSP,推到 redroid 镜像
```

之后每次启动 redroid 实例,挂载不同的 `profile.json`,就是一台"全新设备"。

### 6.5 优劣分析

**优点**:
- 不用买物理真机
- 弹性扩缩容
- 真机环境,易盾自然能跑

**缺点**:
- 厂商云手机指纹高度雷同(同一批 ROM 编出来的)
- 自建 Redroid 改 AOSP 工作量大(等同于做一个改机框架)
- GPU 资源贵,5 台 redroid ≈ 1 台真机价格
- 传感器数据是模拟的,易盾"传感器静止度"判定容易翻车

## 第七章 方案 D:混合架构(实战最优)

### 7.1 核心理念

> **用真机的"质量"换协议的"数量"。**

真正在赚钱的玩家用的都是这个:

```
┌── 协议爬虫(高并发,无真机)─────────────────────┐
│  - 浏览/采集类操作(不敏感)                     │
│  - 占总请求 80%                                 │
│  - 用 sekiro 池里取的 token 当"通行证"          │
└─────────────────────────────┬───────────────────┘
                              │
              敏感操作(注册/发布/支付)
                              ▼
┌── Sekiro 真机签名池(10~50 台真机)──────────────┐
│  - 只用来出 deviceToken / 关键 sign             │
│  - 每次出签名前自动改机                          │
│  - 真机集中托管                                  │
└──────────────────────────────────────────────────┘
                              │
              出现高风险账号时
                              ▼
┌── Redroid 兜底集群(随时可扩)───────────────────┐
│  - 应对突发反风控升级                            │
│  - 当真机池不够用时弹性补位                      │
└──────────────────────────────────────────────────┘
```

### 7.2 三层分工详解

**第一层 — 协议爬虫(80% 流量)**

| 业务 | 是否上协议 | 备注 |
|---|---|---|
| 浏览商品列表 | ✅ | 几乎不查风控 |
| 查看商品详情 | ✅ | 同上 |
| 关注用户 | ✅ | 弱校验 |
| 点赞/收藏 | ✅ | 弱校验 |
| 浏览记录上报 | ✅ | 必须模拟,否则账号"无浏览历史"反而异常 |
| 心跳/在线时长 | ✅ | 必须模拟 |

**第二层 — 真机 Sekiro 池(15% 流量)**

| 业务 | 是否要真机 | 备注 |
|---|---|---|
| 注册账号 | ✅ | 风控最严,必须真机 |
| 实名认证 | ✅ | 同上 |
| 发布商品 | ✅ | 中级风控 |
| 修改商品价格 | ✅ | 中级风控 |
| 私信沟通 | ✅ | 中级风控 |
| 支付/收款 | ✅ | 必须真机 |
| 提现 | ✅ | 必须真机 |

**第三层 — Redroid 兜底(5% 流量)**

| 场景 | 用法 |
|---|---|
| 真机集体被封 | 临时切到 Redroid 维持业务 |
| 测试新机型画像 | Redroid 验证完了再上真机 |
| 跨地域账号 | 不同区域起 Redroid 实例 |

### 7.3 实施路线图

如果你已有 5 台真机,建议路线:

```
Week 1: Sekiro 部署 + 真机注入

Week 4: 全链路联调

   - 上小流量(10 个账号)

Week 5+: 扩量
   - 50 → 100 → 500 账号
```

### 8.6 Profile 生命周期管理

```python
class ProfilePool:
    def acquire(self, account_id):
        """绑定一个 fresh profile 给账号"""
        p = self.db.find_one({"status": "fresh"})
        p["status"] = "in_use"
        p["bound_account"] = account_id
        p["last_used_at"] = time.time()
        self.db.update(p)
        return p
    
    def release(self, profile_id, success):
        if success:
            p = self.db.get(profile_id)
            p["use_count"] += 1
            if p["use_count"] >= 3:  # 用 3 次就废

            self.db.update(p)

            self.db.update_status(profile_id, "burned")
```

## 第九章 协议方案落地的 7 大纪律

### 9.1 一号一画像
- 每个账号绑定唯一 device profile
- 绝不跨账号复用 profile
- profile 失败/封号后立即标记为废弃

### 9.2 一号一 IP
- 每个 profile 绑定独立代理出口
- IP 归属地与 profile 声称的运营商/地区一致
- 同一出口 IP 长期只服务 1-3 个账号

### 9.3 行为节奏模拟
```
注册后操作时序示例:
  T+0:    注册成功
  T+5min: 浏览首页 30 秒
  T+10min: 关注 1 个用户
  T+30min: 退出 App
  T+8h:   再次打开,浏览 5 分钟
  T+1d:   编辑个人资料
  T+3d:   首次发布商品
  T+5d:   首次回复私信
  T+7d:   首次成交
```
**绝不要**注册当天就发布,连续在线 24 小时,毫秒级精准操作。

### 9.4 时间字段不要造假成"完美值"
```
错: boot_time = 0
错: app_install_time = current_time
错: 所有 profile 的 boot_time 都是同一个值

对: boot_time = NOW - random(2h, 72h)
对: 同一批 profile 的时间字段相互错开
```

### 9.5 失败重试不要硬怼
- 一次失败 = 这个 profile 暴露,**直接废弃换新**
- **不要**用原 profile 重试(风控会记账)
- 多次失败 = 整个 IP 段可能进黑名单,换 IP

### 9.6 心跳上报必须有
易盾 SDK 不只在请求时上报,还有定期心跳。协议方案要模拟:
```python
async def heartbeat_loop(account):
    while account.online:
        await asyncio.sleep(random.randint(300, 600))
        await call_yidun_heartbeat(account)
```

### 9.7 持续维护机型库
- 新机型上市后 1-2 个月,易盾才有它的指纹特征
- 永远跟着主流市场跑
- 每月淘汰 2 年以上的老机型

### 10.2 核心 sekiro_client.py

```python
import requests
from typing import Any, Dict
import logging
import time

logger = logging.getLogger(__name__)

class SekiroClient:
    def __init__(self, host: str, group: str, timeout: int = 10):
        self.host = host.rstrip("/")
        self.group = group
        self.timeout = timeout
    
    def invoke(self, action: str, **kwargs) -> Dict[str, Any]:
        url = f"{self.host}/business/invoke"
        params = {"group": self.group, "action": action, **kwargs}
        
        for attempt in range(3):
            try:
                r = requests.post(url, params=params, timeout=self.timeout)
                j = r.json()
                if j.get("status") == 0:
                    return j["data"]
                logger.warning(f"sekiro {action} returned non-zero: {j}")
            except Exception as e:
                logger.warning(f"sekiro {action} attempt {attempt}: {e}")
                time.sleep(0.5 * (attempt + 1))
        raise RuntimeError(f"sekiro {action} failed after 3 attempts")
    
    def get_yidun_token(self) -> str:
        return self.invoke("get_yidun_token")["deviceToken"]
    
    def xianyu_sign(self, api_name: str, data: str) -> str:
        return self.invoke("xianyu_mtop_sign", apiName=api_name, data=data)["sign"]
    
    def heartbeat(self) -> bool:
        return self.invoke("yidun_heartbeat").get("ok", False)
```

### 10.3 profile_pool.py

```python
import json
import random
from pathlib import Path
from typing import Optional, Dict
import threading
import time

class ProfilePool:
    def __init__(self, db_path: Path):
        self.db_path = db_path
        self.lock = threading.Lock()
        self._load()
    
    def _load(self):
        with open(self.db_path, "r", encoding="utf-8") as f:
            self.profiles = json.load(f)
    
    def _save(self):
        with open(self.db_path, "w", encoding="utf-8") as f:
            json.dump(self.profiles, f, ensure_ascii=False, indent=2)
    
    def acquire(self, account_id: str, region: str = None) -> Optional[Dict]:
        with self.lock:
            for p in self.profiles:
                if p["_meta"]["status"] != "fresh":
                    continue
                if region and p.get("ip_binding", {}).get("expected_region") != region:
                    continue
                p["_meta"]["status"] = "in_use"
                p["_meta"]["bound_account"] = account_id
                p["_meta"]["last_used_at"] = int(time.time())
                self._save()
                return p
            return None
    
    def burn(self, profile_id: str):
        """标记 profile 为废弃(失败后)"""
        with self.lock:
            for p in self.profiles:
                if p["_meta"]["profile_id"] == profile_id:
                    p["_meta"]["status"] = "burned"
                    self._save()
                    return
    
    def release(self, profile_id: str, success: bool = True):
        with self.lock:
            for p in self.profiles:
                if p["_meta"]["profile_id"] == profile_id:
                    if success:
                        p["_meta"]["use_count"] = p["_meta"].get("use_count", 0) + 1
                        if p["_meta"]["use_count"] >= 3:
                            p["_meta"]["status"] = "retired"
                        else:
                            p["_meta"]["status"] = "fresh"
                    else:
                        p["_meta"]["status"] = "burned"
                    self._save()
                    return
```

```python
import random

HOUR_WEIGHT = {
    0: 0.1, 1: 0.05, 2: 0.02, 3: 0.01, 4: 0.01, 5: 0.02,
    6: 0.1, 7: 0.3, 8: 0.5, 9: 0.7, 10: 0.8, 11: 0.85,

def should_act_now() -> bool:

def random_action_gap() -> float:
    """两次操作之间的间隔(秒)"""
    # 80% 概率 5-30 秒,20% 概率 1-10 分钟(模拟"看一眼")
    if random.random() < 0.8:
        return random.uniform(5, 30)
    return random.uniform(60, 600)

def random_session_duration() -> int:
    """单次会话时长(秒)"""
    # 平均 3-10 分钟,符合刷闲鱼习惯
    return int(random.gauss(300, 120))
```

### 10.6 主流程示例

```python
def publish_item_flow(account_id: str, item_data: dict):
    pool = ProfilePool(Path("data/profiles.json"))
    sekiro = SekiroClient("http://sekiro:5620", "yidun-pool")
    
    # 1. 获取一个 fresh profile
    profile = pool.acquire(account_id, region="浙江-杭州")
    if not profile:
        raise RuntimeError("no fresh profile available")
    
    profile_id = profile["_meta"]["profile_id"]
    
    try:
        # 2. 模拟自然进入流程: 先浏览
        browse_homepage(profile, account_id, sekiro)
        time.sleep(random_action_gap())
        
        browse_my_items(profile, account_id, sekiro)
        time.sleep(random_action_gap())
        
        # 3. 进入发布页(协议层只是请求,模拟用户停留)
        enter_publish_page(profile, account_id, sekiro)
        time.sleep(random.uniform(30, 90))   # 模拟用户编辑时长
        
        # 4. 真正发布
        result = call_publish_api(profile, account_id, sekiro, item_data)
        
        # 5. 心跳维持
        sekiro.heartbeat()
        
        pool.release(profile_id, success=True)
        return result
    
    except Exception as e:
        # 6. 失败立即废弃 profile
        pool.burn(profile_id)
        raise
```

## 第十一章 监控、告警与失败处置

### 11.1 必须监控的指标

| 指标 | 阈值 | 告警 |
|---|---|---|
| profile 燃烧率 | > 30%/天 | 立即 |
| 单 IP 请求成功率 | < 80% | 立即 |
| sekiro 真机响应延迟 | > 5s | 立即 |
| 单真机签名 QPS | > 5/s | 警告(可能过载) |
| 账号封禁率 | > 2%/天 | 立即 |
| 易盾返回 deviceToken 重复率 | > 5% | 警告(改机失败) |

### 11.2 失败分级处置

```
L1 单次请求失败  → 该 profile 标记 burned, 换 profile 重试
L2 同一 profile 连续 2 次失败 → 检查 sekiro 真机状态
L3 同一 IP 连续 3 个 profile 失败 → IP 进黑名单 30 分钟
L4 同一真机 5 分钟内 3 次出错 token → 重启该真机 + 改机
L5 整体成功率 < 50% 持续 30 分钟 → 全停业,人工介入
```

### 11.3 黑名单与冷却

```python
class IPBlacklist:
    def __init__(self):
        self.banned = {}  # ip -> unban_at
    
    def ban(self, ip, minutes=30):
        self.banned[ip] = time.time() + minutes * 60
    
    def is_banned(self, ip):
        t = self.banned.get(ip)
        return t and t > time.time()
```

## 第十二章 常见踩坑与排错

### 12.1 易盾 deviceToken 拿到但业务请求 401

**原因**: token 与请求头里其他字段(imei/utdid)不一致  
**解决**: 同一 profile 内的所有 ID 必须**同一时刻取**,不能跨调用混用

### 12.2 profile 用第二次就被封

**原因**: 第一次操作后,易盾后端记录了这个"设备指纹"  
**解决**: 严格执行"用一次就废"

### 12.3 注册成功但发布商品失败

**原因**: 注册风控弱,发布风控强 — profile 在发布环节才暴露  
**解决**:
- 用更"成熟"的 profile(boot_time 久,首装时间久)
- 注册到发布间隔 ≥ 24 小时
- 中间穿插浏览/关注等"养号"动作

### 12.4 sekiro 真机签名速度突然变慢

**原因**:
- 真机内存不足
- 易盾 SDK 触发了"频次保护"
- 真机系统升级后改机模块失效

**解决**:
- 监控真机 CPU/内存
- 单机 QPS 限制 < 0.5/s
- 定期检查改机效果(用 DevCheck 比对)

### 12.5 心跳停了账号还是被封

**原因**: 心跳之外还有"前台时长"字段  
**解决**: 协议侧多模拟一个"前台事件上报"

### 12.6 同一台真机出的 token,第 N 次开始重复

**原因**: 改机模块缓存没清干净 → 易盾认为是同一设备  
**解决**: 每次改机后必做:
```bash
pm clear com.taobao.idlefish
rm -rf /sdcard/.um /sdcard/.mobsec ...
am start ...   # 等 30 秒让 SDK 重采集
```

### 12.7 IP 是浙江,但易盾判定异地

**原因**: 你伪造的 SIM IMSI 是云南运营商  
**解决**: IMSI 前 5 位必须与 IP 归属地匹配
```
46000 移动 - 全国
46001 联通 - 全国
46003 电信 - 全国
46011 电信 - 部分地区限定
```

### 12.8 redroid 集群所有实例同一指纹

**原因**: 用了同一个镜像,内置 ANDROID_ID 相同  
**解决**: 每个实例启动时用脚本写入独立 profile.json,改 AOSP 让 framework 读这个
